🎁 お詫び・プレゼント配布システム 運用マニュアル

ビルド（アプリアップデート）を通さずに、GitHub上の announcements.json を更新するだけで、ユーザーにAIクレジットを配布する方法を解説します。

1. 配布の基本ルール

  • アプリは起動時にJSONを読み込み、「未読のお知らせ」を上から順にスキャンします。
  • giftCredits が設定されている場合、対象ユーザーへ自動的にクレジットを加算します。
  • 一度加算されたお知らせは、既読にする前でもDBに記録されるため、再起動しても二重配布されません。
⚠️ 最重要：最新のお知らせは必ずリストの「一番上」に追加してください
アプリは上から順に読み、前回保存した最新のID（最後に読んだID）に到達した時点でスキャンを終了します。古い情報の ID を書き換えても反応しません。

2. JSONの記述例

パターンA：全員に配布（お詫びなど）

[
  {
    "id": "20260320_apology_01",
    "title": "【重要】同期不具合のお詫び",
    "body": "同期が正常に行われない不具合が発生しておりました。お詫びとしてAIクレジットを5つ付与いたしました。",
    "date": "2026/03/20",
    "category": "important",
    "giftCredits": 5,
    "url": ""
  }
]

パターンB：特定のユーザーにのみ配布（個別補填など）

スライドメニューの一番下に表示される8桁のIDを targetIds に記述します。

[
  {
    "id": "20260320_compensation_A",
    "title": "個別補填のお知らせ",
    "body": "不具合のご報告ありがとうございます。補填の10クレジットを付与しました。",
    "date": "2026/03/20",
    "category": "info",
    "giftCredits": 10,
    "targetIds": ["A1B2C3D4"]
  }
]

3. 制限事項と安全機能

🛡️ 二重配布の防止
お知らせポップアップを閉じる前にアプリを落としたとしても、クレジット付与自体は「付与した瞬間」にDB（Metaテーブル）へ LAST_GIFT_PROCESSED_ID として記録されます。そのため、同じお詫びを二度受け取ることはありません。
📊 合計10クレジットの上限チェック
アプリ内の最大保持数は10回です。現在のクレジット + プレゼント数 が10を超える場合、付与は一旦「保留」されます。ユーザーがクレジットを消費して空きができた後、次回のアプリ起動時に自動で付与が再試行されます。

4. JSON全項目リファレンス

announcements.json の各項目の詳細説明です。

項目名 型 説明
id String 必須。 お知らせを識別する一意のID。日付+連番などを推奨。以前と同じIDを使うと配信されません。
title String 必須。 お知らせのタイトル。ポップアップやリストに太字で表示されます。
body String 必須。 お知らせの本文。お詫びの内容などを簡潔に記載します。
date String 必須。 表示用の日付。形式は自由ですが yyyy/MM/dd 形式を推奨します。
category String 必須。 以下のいずれかを指定：
・info : 青色（一般告知）
・update : 緑色（アップデート）
・important : 赤色（お詫び・重要事項）
giftCredits Number 任意。付与するAIクレジット数。指定しない場合は 0 とみなされます。
targetIds Array 任意。特定のユーザーにのみ配る場合のIDリスト（例: ["ID1", "ID2"]）。
省略または null の場合は全員へ配布されます。
url String 任意。詳細な説明がある場合の外部リンクURL。お知らせをタップするとブラウザで開きます。
